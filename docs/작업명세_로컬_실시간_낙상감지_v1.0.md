# CSI-Guard 로컬 실시간 낙상 감지 연동 작업 명세 v1.0

작성일: 2026-07-13
대상: 가정(HOME) 서비스, 장치 1대(수신기 1) 기준 실시간 낙상 감지 E2E 구축

## 1. 목표와 원칙

- 수집부터 연산, 판정, 화면 표시까지 전 과정을 사용자 PC 로컬에서 수행한다. 외부 서버, 클라우드 DB는 사용하지 않는다.
- 유일한 외부 통신은 낙상 발생 시 휴대폰 푸시 알림 전송이며, 이는 마지막 단계에서 구현한다.
- 기존 목업 대시보드(프론트엔드)는 유지하되, 가정용 장치 1대 범위에서 mock 시뮬레이션을 실제 데이터로 대체한다.
- 아래 5개 단계를 순차적으로, 한 단계씩 완료 확인 후 다음으로 진행한다.

## 2. 현재 자산 및 상태 (2026-07-13 기준)

### 2.1 하드웨어 (검증 완료)

| 항목 | 상태 |
|---|---|
| 송신기 (usbmodem5B7B0334061) | csi_send 1c23e84, 5GHz ch48, 100Hz, MAC 1a:00:00:00:00:00 |
| 수신기 (usbmodem5B7B0323351) | csi_recv 1c23e84, USB 시리얼 921600 baud 프레임 출력 |
| 실측 수신률 | 약 91Hz (5초 표본, 체크섬 통과 기준), RSSI 평균 -46dBm |

두 보드 모두 github.com/jspark9703/esp32c5 저장소 main(1c23e84) 기준으로 플래시됨.

### 2.2 소프트웨어 자산

| 자산 | 위치 | 내용 |
|---|---|---|
| 대시보드 (목업) | 이 저장소 src/ | TanStack Start + React 19, mock-store 시뮬레이션, bun run dev로 실행(포트 8080) |
| 수신 펌웨어 소스 | esp32c5/csi_recv/ | 바이너리 프레임 프로토콜 정의의 원본 |
| 참고 백엔드 | esp32c5/tools/fall_detect/ | FastAPI + 시리얼 파서 + 링버퍼 + WebSocket. 프레임 파서와 서버 골격 재활용 후보. 단, 탐지 로직은 이동분산 임계값 방식으로 본 작업의 딥러닝 모델과 다름 |
| 서비스 모델 | Window3BestModelInference/ | 연구단 제공, 아래 2.3 참조 |

### 2.3 서비스 모델 패키지 요약 (Window3BestModelInference)

- 모델: S3 + PCA-ACF 이중 입력 ResNet18 (DualBranchResNet), 낙상/비낙상 이진 분류
- 체크포인트: weights/best_model.pt (epoch 5, 정규화 상수 mean/std 포함)
- 입력 윈도우: 3초, 스트라이드 0.25초
- 수집 샘플링: 약 166.67Hz (3초 = 500 샘플)
- 입력 피처 2종
  - S3: (224, 224) 이미지형 피처
  - PCA-ACF: (1, 128, 64), lag 0.4초 자기상관 기반
- 판정: softmax 낙상 확률, 임계값 0.468, 이후 5윈도우 다수결(mode5) 후처리
- 검증 성능 (임계값 0.468 + mode5): Macro F1 0.8936, 낙상 Recall 0.75

### 2.4 바이너리 프레임 프로토콜 (수신기 → PC)

- 매직 0xA55A(LE), version 1, frame_type 1(CSI)
- 헤더 46바이트: seq, MAC, RSSI, 채널, 타임스탬프(us, 2^32 wrap), fft_gain, agc_gain, csi_len 등
- 페이로드 최대 612바이트: (허수, 실수) int8 쌍 인터리브, 최대 306 서브캐리어
- 트레일러: 앞 전체 바이트 합의 하위 16비트 체크섬

## 3. 목표 아키텍처

```
[송신기 ESP32-C5] --ESP-NOW 5GHz ch48--> [수신기 ESP32-C5]
                                              |
                                        USB 시리얼 921600
                                              v
                              [로컬 백엔드 (Python, FastAPI)]
                               1. 시리얼 리더 + 프레임 파서
                               2. 링버퍼 (수 초 분량)
                               3. 피처 추출 (S3, PCA-ACF)   <- 단계 3에서 확보/구현
                               4. 모델 추론 (best_model.pt)
                               5. 임계값 + mode5 후처리
                               6. REST + WebSocket 서빙
                                    |                +--> [푸시 알림 (단계 5)]
                                    v
                          [대시보드 (localhost:8080)]
                           가정용 기기 1대 실데이터 표시
```

백엔드는 esp32c5/tools/fall_detect의 구조(시리얼 리더 스레드, 링버퍼, WebSocket 10Hz 푸시)를 골격으로 삼고, 탐지 엔진만 이동분산 임계값에서 딥러닝 모델 추론으로 교체하는 방향을 기본안으로 한다. 구체 위치(신규 디렉토리 여부)는 단계 1 시작 시 결정한다.

## 4. 단계별 작업 명세

### 단계 1. 시리얼 포트 통신 설정 구현

목표: 가정 내 장치 1개(수신기)와 로컬 백엔드 사이의 시리얼 연결 계층을 구현한다.

작업 항목:
1. 수신기 포트 자동 탐지 (macOS cu.usbmodem 패턴, 시리얼 번호 기반 식별 포함)
2. 921600 baud 연결, 연결 상태 관리 (연결, 끊김 감지, 자동 재연결)
3. 바이너리 프레임 파서 구현 (2.4 프로토콜: 매직 스캔, 길이 검증, 체크섬 검증, 재동기화)
4. 타임스탬프 unwrap (2^32 마이크로초 랩어라운드 보정)
5. 파싱 결과의 구조화 (seq, rssi, gain, 서브캐리어 진폭 배열 등)

사전 확인 필요 (구현 전에 함께 결정):
- 백엔드 코드의 위치와 형태 (fall_detect 포크 여부, 신규 디렉토리명)
- 진폭만 쓸지 위상도 보존할지 (모델 피처 정의에 따라 결정, 단계 3 확인과 연동)

완료 기준:
- 수신기 연결 후 프레임 수신률, 체크섬 실패율, RSSI를 콘솔 또는 엔드포인트로 확인 가능
- USB를 뽑았다 꽂아도 자동 복구

### 단계 2. 실시간 데이터 스트림 수신 구현

목표: 파싱된 CSI 프레임을 모델 입력에 적합한 형태로 계속 축적하고 상태를 노출한다.

작업 항목:
1. 링버퍼 구현 (최소 3초 윈도우 + 여유, 목표 5초 이상 분량)
2. 수신 품질 지표 산출 (실효 Hz, 유실률, 마지막 수신 시각)
3. FastAPI 서버 골격 + 상태 REST 엔드포인트 (/monitor/status 등)
4. WebSocket 실시간 채널 (원시 진폭 요약 또는 다운샘플 스트림, 대시보드 디버그용)

완료 기준:
- 브라우저 또는 curl로 실시간 수신 상태(Hz, 버퍼 충전율)를 확인 가능
- 1시간 연속 수신에서 크래시, 메모리 증가 없음

### 단계 3. 서비스 모델 실시간 연동 구현

목표: 링버퍼의 CSI 스트림을 0.25초마다 3초 윈도우로 잘라 모델에 연속 투입한다.

작업 항목:
1. 피처 추출 파이프라인 확보: 원시 CSI에서 S3 (224,224)와 PCA-ACF (1,128,64)를 만드는 코드
2. 체크포인트 로드 (DualBranchResNet, 정규화 상수는 체크포인트 내장값 사용)
3. 0.25초 주기 추론 루프 (macOS는 MPS 가속 시도, 실패 시 CPU)
4. 임계값 0.468 적용 + 실시간용 mode5 후처리 (아래 지연 이슈 참조)
5. 추론 지연 측정 (윈도우 마감부터 판정까지 ms)

중대 갭, 반드시 선결:
- 패키지에 피처 추출 코드가 없음. infer_validation.py는 미리 계산된 npz를 읽기만 하며, 원시 CSI CSV에서 S3와 PCA-ACF를 생성한 코드는 포함되어 있지 않다. 연구단에서 받거나 정의를 받아 재구현해야 한다.
- 샘플링 주파수 불일치. 모델은 166.67Hz 수집 데이터(3초 = 500샘플, 유효 대역 최대 83.3Hz)로 학습되었는데 현재 송신기는 100Hz(나이퀴스트 50Hz)로 송신 중이다. 100Hz 데이터를 보간해도 50에서 83.3Hz 대역 정보는 만들어낼 수 없으므로, 송신 주파수를 200Hz로 되돌리는 것(과거 플래시본과 동일)이 필요할 가능성이 높다. 연구단이 수집 당시 사용한 송신 주파수 확인 필요.
- mode5 후처리의 지연. 중심 윈도우 기준 다수결이므로 미래 윈도우 2개가 필요, 판정 확정에 약 0.5초 추가 지연이 발생한다. 실시간에서는 인과(causal) 다수결로 바꿀지, 지연을 수용할지 결정 필요.

연구단 확인 질문 목록:
1. 원시 CSI에서 S3, PCA-ACF 피처를 생성하는 코드 또는 정확한 정의(전처리, 서브캐리어 선택 규칙 selected_stream_indices, PCA 학습 방식과 고정 여부, ACF lag 구성)를 받을 수 있는가
2. 수집 당시 송신기 설정(송신 주파수, 채널, 대역폭)은 무엇이었나. 166.67Hz는 어떻게 형성된 값인가 (200Hz 송신 후 리샘플인지)
3. 실시간 운영 시 권장 임계값과 후처리 방식 (0.468 + mode5를 그대로 쓰면 되는지)
4. 진폭만 사용하는지, 위상 정보도 사용하는지

완료 기준:
- 실보드 스트림에 대해 0.25초마다 낙상 확률이 산출되고 로그로 확인 가능
- 추론 1회 지연이 스트라이드(250ms) 이내

### 단계 4. 낙상/비낙상 결과 표시 구현

목표: 대시보드의 가정용 화면에서 mock 대신 실제 판정 결과를 표시한다.

작업 항목:
1. 백엔드 WebSocket/REST 스키마 확정 (낙상 확률, 판정 상태, 수신 품질)
2. 프론트엔드 데이터 계층 연결: mock-store의 시뮬레이션 tick을 유지한 채, 실장치 1대에 한해 백엔드 구독으로 대체 (기존 FallAlarmModal, 상태 표기 IDLE 대기, FALL 낙상 등 도메인 용어 재사용)
3. 낙상 발생 시 알람 모달 + 이벤트 로그 기록 (기존 UI 흐름 재사용)
4. 백엔드 미가동, 장치 미연결 시의 표시 (연결 끊김 상태)

완료 기준:
- 실제로 사람이 움직이거나 낙상 시연 시 대시보드에 확률과 판정이 실시간 반영
- 백엔드 종료 시 대시보드가 오류 없이 연결 끊김 표시

### 단계 5. 낙상 시 휴대폰 푸시 알림 구현 (추후)

목표: FALL 판정 확정 시 휴대폰으로 푸시 알림을 보낸다.

후보 방식 (구현 시점에 결정):
1. ntfy (셀프호스트 또는 ntfy.sh, 앱 설치, 로컬 우선 원칙에 가장 부합)
2. FCM (Firebase Cloud Messaging, 자체 앱 또는 PWA 필요)
3. 카카오 알림톡, SMS 게이트웨이 (비용 발생, 국내 친화)

작업 항목 (개요만):
1. 판정 확정 조건 정의 (연속 낙상 윈도우 수, 쿨다운)
2. 알림 발송 모듈 + 재시도
3. 대시보드 알림 설정 화면 연동 (기존 notifications 라우트 재사용)

완료 기준:
- 낙상 시연 후 수 초 내 휴대폰 알림 수신

## 6. 진행 방식

- 단계 1부터 순서대로 진행하며, 각 단계 완료 기준을 함께 확인한 뒤 다음 단계로 넘어간다.
- 단계 3의 연구단 질문(5절 1, 2, 5번 리스크)은 단계 1, 2 진행과 병행하여 미리 발송해 두는 것을 권장한다.
- 이 문서는 각 단계 착수 시 세부 설계를 추가하며 버전을 올린다.
